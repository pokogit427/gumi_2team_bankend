# 11_백엔드_배포계획

이 문서는 LocalHub 백엔드의 Render 배포 방식을 정리한 문서다. 구현 전 기준으로 작성되었으며, 실제 배포 시 참고한다.

---

## 1. 배포 목표

- FastAPI 백엔드를 Render에 배포한다.
- 프론트엔드와 백엔드가 별도로 동작하도록 분리한다.
- 환경변수와 SQLite 저장 경로를 명확히 관리한다.
- 배포 후 기본 건강 확인과 Swagger 확인이 가능하도록 한다.

---

## 2. Render 배포 구조

### 권장 구조
- Backend 서비스: Render Web Service
- Frontend 서비스: Netlify
- 데이터 저장소: Render 내부 파일 시스템 + SQLite

### 배포 구성 개념
- 백엔드 서비스는 FastAPI 앱을 실행한다.
- 프론트엔드는 백엔드 API 주소를 환경변수 또는 설정값으로 사용한다.
- 지역정보 JSON 파일은 배포본에 포함된다.

---

## 3. Git 저장소 연결 방식

### 방식
- GitHub 저장소를 Render에 연결한다.
- main 또는 dev 브랜치 기준으로 자동 배포되도록 설정한다.

### 권장 방식
- 코드 변경 시 GitHub에 push한다.
- Render에서 자동 배포를 실행한다.
- 수동 재배포도 가능하다.

---

## 4. Build Command

Render에서 사용하는 Build Command는 다음과 같이 설정한다.

```bash
pip install -r requirements.txt
```

### 이유
- 의존성 패키지를 설치해야 FastAPI 실행이 가능하다.

---

## 5. Start Command

Render에서 사용하는 Start Command는 다음과 같이 설정한다.

```bash
uvicorn app.main:app --host 0.0.0.0 --port 10000
```

### 참고
- Render는 기본적으로 포트 번호를 환경변수로 제공하므로, 필요 시 `$PORT`를 사용하도록 조정할 수 있다.

---

## 6. 환경변수 목록

다음 환경변수를 설정한다.

| 변수명 | 설명 |
|---|---|
| DATABASE_URL | SQLite 경로 또는 DB 연결 문자열 |
| CORS_ORIGINS | 허용할 프론트엔드 Origin 목록 |
| ENV | 실행 환경 (development, production) |
| DEBUG | 디버그 모드 여부 |

### 권장 값
- `DATABASE_URL=sqlite:///./app.db`
- `CORS_ORIGINS=https://example.netlify.app,http://localhost:5173`
- `ENV=production`
- `DEBUG=false`

---

## 7. data 폴더 JSON 포함 여부

### 포함 여부
- 배포 시 data 폴더의 JSON 파일을 포함한다.
- 백엔드 코드에서 해당 파일을 읽을 수 있어야 한다.

### 권장 방식
- 코드 기준으로 `data/` 경로를 상대 경로로 참조한다.
- 배포 환경에서도 파일이 존재하는 구조로 유지한다.

---

## 8. JSON 파일 경로 처리 방식

### 권장 방식
- 실행 디렉터리 기준으로 경로를 해석한다.
- 가능한 경우 `Path(__file__).resolve().parent.parent / "data"` 형태로 처리한다.
- 절대 경로를 하드코딩하지 않는다.

### 이유
- 로컬 실행과 Render 배포 환경에서 경로 불일치가 발생하지 않도록 하기 위함이다.

---

## 9. SQLite 파일 경로

### 권장 방식
- SQLite 파일은 프로젝트 루트 아래 `app.db` 또는 `data/app.db`에 저장한다.
- 배포 환경에서는 파일이 컨테이너 재시작 시 유지되는지 확인해야 한다.

### 권장 예시
- `sqlite:///./app.db`

---

## 10. Render에서 SQLite 영속성 위험

Render의 Web Service는 컨테이너 재배포 또는 재시작 시 파일 시스템이 초기화될 수 있어 SQLite 파일이 사라질 수 있다.

### 위험 요소
- 배포 재시작 시 로컬 파일이 사라질 수 있다.
- 데이터 유실 가능성이 있다.

### 대응 방안
- MVP 단계에서는 허용하되, 운영 단계에서는 PostgreSQL로 이전하는 것을 권장한다.
- SQLite는 개발/데모 목적에 적합하다.
- 중요한 데이터는 반드시 백업한다.

---

## 11. 배포 후 /health 확인

배포 완료 후 다음을 확인한다.

- `/health` 엔드포인트가 200 응답을 반환한다.
- 서버가 정상적으로 실행 중임을 확인한다.

### 기대 결과
- `{"status":"ok"}` 형태 또는 유사한 응답

---

## 12. Swagger 확인

배포 후 다음 URL을 확인한다.

- `/docs`
- `/redoc`

### 확인 포인트
- Swagger UI가 정상적으로 로드된다.
- API 목록이 표시된다.

---

## 13. Netlify Origin CORS 등록 방법

프론트엔드가 Netlify에 배포된 경우, 백엔드의 CORS 허용 목록에 Netlify Origin을 추가한다.

### 등록 예시
- `https://your-app-name.netlify.app`
- `http://localhost:5173` (개발용)

### 권장 방식
- 환경변수 `CORS_ORIGINS`에 쉼표로 구분해 등록한다.
- 백엔드 재배포 후 CORS 설정이 반영된다.

---

## 14. 재배포 절차

1. 코드 변경 후 GitHub에 push한다.
2. Render 대시보드에서 해당 서비스의 Deploy 버튼을 누른다.
3. Build 로그를 확인한다.
4. 배포가 완료되면 `/health`와 `/docs`를 확인한다.
5. 프론트엔드에서 API 호출이 정상 동작하는지 확인한다.

---

## 15. 롤백 방법

### 빠른 롤백
- 이전 커밋으로 되돌린 뒤 다시 push한다.
- Render에서 이전 배포로 재배포한다.

### 대안
- Render의 이전 배포 기록을 이용해 이전 버전으로 되돌린다.

### 주의사항
- 데이터베이스 파일이 사라질 수 있으므로, 롤백 전 데이터 백업을 확인한다.

---

## 16. 배포 완료 기준

다음 조건을 모두 충족하면 배포를 완료한 것으로 판단한다.

- Render에서 백엔드 서비스가 정상 실행 중이다.
- `/health`가 정상 응답한다.
- `/docs`가 정상 접근된다.
- 프론트엔드 Origin에 대해 CORS가 허용된다.
- JSON 파일 경로가 배포 환경에서도 정상 동작한다.
- 기본 CRUD, 검색, 챗봇 API가 테스트된다.

---

## 17. 배포 시 주의사항

- `.env` 파일은 Git에 포함하지 않는다.
- 민감정보는 Render 환경변수로 관리한다.
- SQLite 데이터는 운영 환경에서 중요한 데이터로 취급하지 않는다.
- 배포 전 환경변수 값과 경로를 반드시 확인한다.
