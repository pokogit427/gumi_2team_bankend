| 구분 | 항목 | 상세내용 | 근거 조항 | 검토 의견 |

| 필수 | 서비스 범위 및 권역 선정 | 서울, 대전/충청, 구미/경북, 광주/전라, 부산 중 1개 권역을 선정하여 지역 주민과 관광객이 정보를 공유·소비하는 익명 커뮤니티 LocalHub를 구축한다. | II-2 (p.1) | 실제 제공 JSON의 데이터량·좌표·카테고리 완성도를 비교해 착수 전에 권역을 확정해야 한다. 권역 변경은 데이터 매핑과 화면 구성을 다시 해야 하므로 일정 위험이 크다. |
| 필수 | 제공 JSON 기반 데이터 연동 | 고객사가 사전 수집·가공한 JSON 파일을 기준으로 Vue.js 화면과 FastAPI 서버의 데이터 연동을 구현한다. 직접 공공 API 호출은 요구되지 않는다. | III-1-가 (p.1) | 현재 첨부에는 실제 JSON 파일이 없어 필드명, 인코딩, 좌표 형식, 중복·결측값을 확인할 수 없다. JSON 스키마 확인을 개발 착수 조건으로 둔다. |
| 필수 | 익명 커뮤니티 운영 | 회원가입과 로그인 없이 운영하며 사용자 인증, 역할, 권한 체계를 적용하지 않는다. | III-2-가 (p.2) | 개인정보 입력·수집 기능을 임의로 추가하지 않는다. 익명 사용자 식별이나 세션 관리가 필요하다면 RFP 범위를 넘지 않는 최소 수준으로 제한한다. |
| 필수 | 게시글 작성 항목 | 게시글 작성 시 제목, 내용, 수정용 비밀번호를 함께 입력·등록한다. | III-2-나 (p.2) | 프론트엔드와 백엔드 양쪽에서 필수값 검증이 필요하다. 제목·본문·비밀번호의 길이 제한은 문서에 없으므로 팀 기준을 사전에 정해야 한다. |
| 필수 | 비밀번호 기반 수정·삭제 | 게시글 수정·삭제 요청 시 등록된 수정용 비밀번호와 일치하는지만 확인하여 권한을 판단한다. | III-2-나 (p.2) | 불일치 응답에서 저장 비밀번호가 노출되지 않도록 한다. 수정·삭제 API의 비밀번호 전달 방식과 오류 코드를 기능 명세서에 정의한다. |
| 필수 | 카테고리 게시판 CRUD | 선정 권역의 카테고리 게시판에 대해 목록 조회, 상세 조회, 작성, 수정, 삭제 FastAPI REST API와 Vue.js 화면을 구현한다. | III-2-다 (p.2) | 검색과 페이지네이션은 예시 화면에 있으나 선택 기능 표에도 게시글 검색이 포함되어 필수 여부가 모호하다. 기본 목록·상세·CRUD를 우선하고 검색은 선택 기능으로 관리한다. |
| 필수 | 챗봇 API 엔드포인트 | FastAPI POST /api/chat 엔드포인트를 구현한다. | III-3-가 (p.2) | 요청·응답 JSON 스키마, 오류 응답, 타임아웃, 빈 질문 처리 기준을 먼저 정해야 프론트엔드와 병렬 개발이 가능하다. |
| 필수 | 제공 데이터 기반 챗봇 응답 | 제공 JSON 데이터를 근거로 자연어 지역 정보 질의에 응답하도록 구현한다. | III-3-가 (p.2) | 전체 JSON을 매 요청마다 전달하기보다 카테고리·키워드 필터링 또는 검색 후 필요한 데이터만 컨텍스트로 구성하는 방식이 비용과 응답 속도에 유리하다. |
| 필수 | 챗봇 주요 질의 유형 | 관광지 추천, 축제 일정, 모범음식점 위치, 커뮤니티 게시글 검색 등 주요 질의 유형에 응답한다. | III-3-나 (p.2) | 챗봇의 커뮤니티 게시글 검색은 필수 질의 유형으로 해석한다. 게시판 자체 검색 UI가 선택 기능인 것과 구분해 API 범위를 설계해야 한다. |
| 필수 | 채팅 UI 및 모바일 대응 | 대화 히스토리를 유지하는 채팅 UI를 개발하고 플로팅 형태와 모바일 대응을 제공한다. 모바일에서는 펼친 챗봇을 전체 화면으로 표시한다. | III-3-다, IV-참고4-⑤ (p.2, p.8) | 히스토리 유지 범위가 현재 세션인지 브라우저 재접속 후까지인지 미정이다. MVP에서는 프론트 상태 또는 sessionStorage 기준을 명시하는 것이 현실적이다. |
| 필수 | 프론트엔드 Vue.js 3 SPA | 프론트엔드는 Vue.js 3을 사용하고 SPA 구조로 구성한다. | III-4-가 (p.2) | 라우팅, API 클라이언트, 상태 관리 방식은 팀이 정할 수 있으나 프레임워크와 SPA 조건은 변경하지 않는다. |
| 필수 | 백엔드 FastAPI REST | FastAPI 기반 REST API 서버를 구축한다. | III-5-가 (p.2) | 게시판, 챗봇, 지역 데이터 API의 URL 규칙과 응답 포맷을 초기에 통일해야 한다. 자동 API 문서로 연동 테스트를 빠르게 진행한다. |
| 필수 | SQLAlchemy 및 SQLite 스키마 | SQLAlchemy ORM을 적용하여 SQLite 스키마를 설계한다. posts, comments, locations 등은 예시 엔터티이다. | III-5-가 (p.2) | comments는 예시일 뿐 댓글 기능이 필수라는 의미는 아니다. 필수 범위에 필요한 posts와 지역 데이터 구조부터 최소 스키마로 확정한다. |
| 필수 | 활용 데이터 목록 문서화 | 기능 명세서에 활용 데이터의 출처, 라이선스, 수집일을 별도 항목으로 정리한다. | III-1-다, IV-참고1 (p.2~3) | 고객사가 제공한 JSON도 출처·라이선스 메타정보가 필요하다. 해당 정보가 함께 제공되지 않으면 고객사에 확인 요청해야 한다. |
| 필수 | 지정 플랫폼 배포 | 프론트엔드는 Netlify, 백엔드는 Render에 배포한다. | III-6-가~나 (p.2) | CORS, 환경변수, 빌드 명령, 배포 브랜치를 조기에 검증한다. Render 환경에서 SQLite 파일이 재배포·재시작 후 유지되는지도 사전 시험이 필요하다. |
| 필수 | 배포 URL 동작 검증 | 배포 완료 후 Netlify 사이트와 Render API의 외부 접근 URL이 정상 동작하는지 확인하고 산출물에 포함한다. | III-6-다, IV-참고1 (p.2~3) | 프론트에서 실제 배포 API 호출, 새로고침 라우팅, CRUD, 챗봇까지 운영 URL 기준으로 최종 점검한다. |
| 필수 | 지정 산출물 패키지 | 프론트엔드 저장소 URL, 백엔드 저장소 URL, 초기 데이터가 포함된 SQLite .db 파일, Netlify URL, Render URL, 기능 명세서, WBS, 발표 자료를 제출한다. | IV-참고1 (p.3) | 제출 직전 확인이 아니라 담당자와 완료 기준을 WBS에 각각 배정한다. 저장소 URL과 배포 URL은 외부 접근 권한도 확인한다. |
| 필수 | 선택 기능 최소 1개 구현 | 모든 필수 기능을 완료한 후 선택 기능을 최소 1개 이상 구현한다. | IV-참고2 (p.3) | 납기와 팀 숙련도를 고려해 외부 API 의존성이 낮은 기능 1개를 우선 확정한다. 여러 기능을 동시에 착수하면 핵심 기능 완성도가 낮아질 수 있다. |
| 필수 | 기술 검토 및 일정 초안 | 다음 미팅 시 기술 검토 의견과 개발 일정 초안을 준비하여 참석한다. | V-6 (p.8) | 확인 필요사항, 기술 위험, 역할 분담, 의존 작업을 포함한 1차 WBS를 준비하면 착수 후 범위 논쟁을 줄일 수 있다. |
| 선택 | OpenAI API 활용 | 챗봇 구현에 OpenAI API를 활용할 수 있다. | III-3-가 (p.2) | 사용 모델, 요청 한도, 키의 유효기간과 비용 한도가 문서에 없다. 제공 키의 실제 사용 가능 범위를 먼저 확인한다. |
| 선택 | 지도 시각화 | Leaflet.js 또는 Kakao Maps 등을 이용해 관광지·맛집을 지도 핀으로 표시하고 권역 필터를 제공한다. | IV-참고2 (p.3~4) | 좌표 데이터가 충분하다면 사용자 체감 효과가 크다. 지도 API 키와 이용약관, 호출량을 확인해야 한다. |
| 선택 | 데이터 시각화 대시보드 | Chart.js 또는 D3.js로 게시글 현황과 인기 지역 통계를 시각화한다. | IV-참고2 (p.4) | 외부 API가 없어 일정 위험이 낮다. 다만 초기 게시글이 적으면 시각화 의미가 약하므로 샘플 데이터 기준을 정한다. |
| 선택 | 축제 캘린더 | 권역별 축제 일정을 캘린더 형태로 시각화한다. | IV-참고2 (p.4) | 축제 데이터에 시작일·종료일이 일관되게 존재하는지 확인한다. FullCalendar 적용 시 모바일 레이아웃을 함께 검증한다. |
| 선택 | 경로 안내 | 복수의 관광지·맛집을 선택하여 이동 경로를 지도 위에 시각화한다. | IV-참고2 (p.4) | 지도와 경로 API가 모두 필요할 수 있어 비용·키·라이선스 의존성이 높다. 짧은 일정에서는 후순위가 적절하다. |
| 선택 | 날씨 정보 연동 | 외부 날씨 API로 권역별 현재 날씨와 여행 적합 여부를 표시한다. | IV-참고2 (p.4) | 추가 API의 이용 조건과 비용 확인이 필요하며, 여행 적합 여부의 판단 규칙을 명확히 정의해야 한다. |
| 선택 | 실시간 알림 | WebSocket으로 새 게시글 알림 또는 접속자 현황을 표시한다. | IV-참고2 (p.4) | 배포 환경의 WebSocket 지원과 연결 안정성 검증이 필요하다. 핵심 CRUD보다 테스트 범위가 넓어 일정 위험이 높다. |
| 선택 | 다국어 지원 | i18n을 활용하여 한/영 전환과 다국어 지역 정보 콘텐츠를 제공한다. | IV-참고2 (p.4) | UI 번역만 할지 데이터와 챗봇 응답까지 포함할지 범위를 확정해야 한다. JSON 원문 언어와 번역 품질도 확인한다. |
| 선택 | 소셜 공유 연동 | KakaoTalk, 링크 복사 등 공유 기능과 OG 태그 메타데이터를 구성한다. | IV-참고2 (p.4) | SPA의 OG 태그 노출 방식과 Kakao 공유 키 설정이 추가로 필요하다. 단순 링크 복사는 저위험 대안이다. |
| 선택 | 커뮤니티 게시판 추가 기능 | 조회수, 게시글 검색, 북마크, 좋아요, 이미지 첨부, 태그 기능 중 필요한 항목을 추가한다. | IV-참고2 (p.4) | 예시 게시판 화면에는 검색이 있으나 선택 기능 목록에도 검색이 기재되어 있다. 검색 UI는 선택으로 두고 구현 여부를 MVP에서 명확히 선언한다. |
| 제약 | 1개 권역으로 범위 제한 | 5개 권역 전체가 아니라 제공 데이터 중 1개 권역만 선정하여 구현한다. | II-2 (p.1) | 여러 권역 동시 지원은 스코프 확대에 해당한다. 공통 구조를 고려하되 실제 데이터·화면은 선정 권역 기준으로 제한한다. |
| 제약 | 추가 데이터 라이선스 사전 확인 | 제공 JSON 외 데이터를 수집할 경우 라이선스와 공공누리 유형 1~4를 반드시 사전에 확인한 뒤 사용한다. | II-2, III-1-나 (p.1) | 확인 기록 없이 데이터를 가져오지 않는다. 출처 URL, 이용 조건, 확인일, 담당자를 남겨야 한다. |
| 제약 | 추가 데이터 활용 가능성 검토 | 추가 데이터가 상업적·서비스 목적에 사용 가능한지 판단한 후 수집·활용한다. | III-1-나 (p.1~2) | 비영리·출처표시·변경금지 조건을 기능과 대조해야 한다. 사용이 불명확하면 제공 JSON만으로 범위를 축소한다. |
| 제약 | API 키 사용 범위 및 비용 상한 | 개발용 API 키는 고객사가 제공하며 외부 서비스는 제공 키의 범위 내에서 운영하고 정해진 예산을 초과할 수 없다. | II-2 (p.1) | 예산 금액과 호출 한도가 문서에 없다. 모델·지도·날씨 API별 한도를 착수 전에 서면 확인하고 사용량을 모니터링한다. |
| 제약 | 기술 스택 및 배포 플랫폼 고정 | Vue.js 3, FastAPI, SQLite와 Netlify·Render 조합을 사용하며 임의의 대체 기술로 변경하지 않는다. | II-2, III-4~6 (p.1~2) | 팀 선호나 편의에 따라 React, Spring, MySQL, 다른 배포 플랫폼으로 변경하면 요구 불충족이다. |
| 제약 | 커뮤니티 데이터 DB 저장 의무 | 커뮤니티 데이터는 반드시 데이터베이스에 저장·관리되어야 한다. | III-2-라 (p.2) | 메모리나 JSON 파일만으로 게시글을 관리하면 불합격이다. SQLite 파일 생성과 CRUD 영속성을 테스트한다. |
| 제약 | 비밀번호 평문 저장 | 수정용 비밀번호는 암호화하지 않고 평문으로 저장·비교한다. 교육 목적의 의도된 설계이다. | III-2-나 (p.2) | 실무 보안 원칙과 반대이므로 과제 외 환경에 재사용하지 않는다. 실제 개인정보나 실제 비밀번호를 입력하지 않도록 안내한다. |

| 제약 | AI 개발도구 허용 범위 | VSCode Copilot과 OpenAI API Key 조합만 허용하며 해당 환경 기준으로 개발·납품한다. | III-7-가 (p.2) | 팀원 전원이 동일 정책을 숙지하고 사용 도구를 작업 시작 전에 확인한다. |

| 제약 | SSAFY GitLab 제출 | 납기 내 결과물과 발표 자료, 지정 산출물을 모두 SSAFY GitLab에 제출한다. | II-3 (p.1) | 저장소 권한, 브랜치, 제출 경로를 미리 확인한다. Git Repository URL의 실제 플랫폼이 GitLab인지 별도 저장소인지 확인이 필요하다. |
