# 9_백엔드_아키텍처

이 문서는 LocalHub 백엔드의 폴더 구조와 기능별 책임을 설계한 문서다. 실제 기능 코드는 아직 구현하지 않으며, 초보자가 이해하고 3일 안에 구현 가능한 최소 구조로 정리했다.

---

## 1. 전체 구조

추천 구조는 다음과 같다.

```text
backend/
├─ app/
│  ├─ __init__.py
│  ├─ main.py
│  ├─ config.py
│  ├─ database.py
│  ├─ models.py
│  ├─ schemas.py
│  ├─ dependencies.py
│  ├─ routers/
│  │  ├─ __init__.py
│  │  ├─ posts.py
│  │  ├─ locations.py
│  │  ├─ search.py
│  │  └─ chat.py
│  └─ services/
│     ├─ __init__.py
│     ├─ post_service.py
│     ├─ location_service.py
│     ├─ search_service.py
│     └─ chat_service.py
├─ tests/
├─ .env.example
├─ .gitignore
├─ requirements.txt
└─ README.md
```

### 선택 이유
- FastAPI의 기본 패턴과 잘 맞는다.
- Router와 Service를 분리해 책임이 명확하다.
- 초보자가 파일 수를 관리하면서도 기능별로 확장하기 쉽다.
- 게시글, 지역정보, 검색, 챗봇 기능을 각각 분리해 구현 순서를 정하기 좋다.

---

## 2. 파일별 책임

| 파일 | 책임 |
|---|---|
| app/main.py | FastAPI 앱 생성, 라우터 등록, CORS 설정 |
| app/config.py | 환경변수 로딩, 설정값 관리 |
| app/database.py | SQLite 연결, Session 생성, 테이블 초기화 |
| app/models.py | SQLAlchemy ORM 모델 정의 |
| app/schemas.py | 요청/응답 Pydantic 스키마 정의 |
| app/dependencies.py | 공통 의존성(세션, 서비스 객체 등) |
| app/routers/posts.py | 게시글 CRUD API 처리 |
| app/routers/locations.py | 지역정보 조회 API 처리 |
| app/routers/search.py | 통합 검색 API 처리 |
| app/routers/chat.py | 챗봇 API 처리 |
| app/services/post_service.py | 게시글 관련 비즈니스 로직 |
| app/services/location_service.py | 지역정보 JSON 로딩 및 조회 로직 |
| app/services/search_service.py | 공공데이터 + 게시글 통합 검색 로직 |
| app/services/chat_service.py | 챗봇 응답 로직 |

---

## 3. Router 역할

Router는 HTTP 요청을 받아서 적절한 Service를 호출하고, 응답을 반환하는 역할을 한다.

### posts Router
- GET /api/posts
- GET /api/posts/{post_id}
- POST /api/posts
- PUT /api/posts/{post_id}
- DELETE /api/posts/{post_id}

### locations Router
- GET /api/locations
- GET /api/locations/{content_id}

### search Router
- GET /api/search

### chat Router
- POST /api/chat

### Router의 원칙
- Router는 비즈니스 로직을 직접 쓰지 않는다.
- Service를 호출해 결과를 받는다.
- 예외 처리는 Service 또는 공통 예외 처리 로직에서 담당한다.

---

## 4. Service 역할

Service는 실제 로직을 처리하는 계층이다.

### post_service.py
- 게시글 생성, 조회, 수정, 삭제
- 비밀번호 검증
- 조회수 증가
- DB 세션과 모델 조작

### location_service.py
- JSON 파일 로드
- 메모리 캐시 관리
- 지역정보 필터링, 검색, 상세 조회
- `mapx`, `mapy` 문자열 -> float 변환

### search_service.py
- 공공데이터 결과와 게시글 결과를 통합
- `query`, `category` 기반 검색
- `result_type` 구분

### chat_service.py
- 사용자 메시지 분류
- 지역정보 또는 게시글 검색 결과를 바탕으로 응답 생성
- `answer`, `references` 형태로 반환

---

## 5. Schema 역할

Schema는 API 요청/응답 형식을 정의하는 역할을 한다.

### 요청 Schema
- 게시글 생성/수정 요청
- 챗봇 요청
- 검색 파라미터

### 응답 Schema
- 게시글 응답
- 지역정보 응답
- 검색 응답
- 챗봇 응답

### 역할 분리
- `models.py`는 DB 구조를 의미한다.
- `schemas.py`는 API 계층의 데이터 형식을 의미한다.

---

## 6. Model 역할

`models.py`에는 SQLAlchemy ORM 모델을 정의한다.

### 현재 설계 기준 모델
- `Post` 모델만 사용한다.
- 지역정보는 ORM 모델이 아니라 JSON 로딩으로 처리한다.

### 모델 정의 기준
- `posts` 테이블과 1:1 대응
- `id`, `title`, `content`, `password`, `view_count`, `created_at`, `updated_at` 포함

---

## 7. Database 역할

`database.py`는 DB 연결과 세션 관리를 담당한다.

### 역할
- SQLite 연결 URL 관리
- SessionLocal 생성
- 테이블 생성
- 세션 종료 처리

### 권장 방식
- `sqlite:///./app.db` 형태로 관리한다.
- 테스트 환경과 개발 환경은 분리할 수 있도록 설정 가능하게 만든다.

---

## 8. Config 역할

`config.py`는 환경변수와 앱 설정을 관리한다.

### 관리 항목
- `DATABASE_URL`
- `CORS_ORIGINS`
- `DEBUG`
- `ENV`

### 이유
- 배포 환경과 로컬 환경의 차이를 최소화하기 위해 필요하다.
- `.env` 파일을 로드해 설정값을 쉽게 바꿀 수 있다.

---

## 9. JSON 로더 역할

지역정보는 별도 테이블 없이 JSON 파일을 읽어 사용한다.

### 로딩 책임
- [data/구미_경북권_관광지.json](../data/구미_경북권_관광지.json) 등 JSON 파일 읽기
- 각 파일의 `items` 배열 로드
- 메모리 캐시 저장
- 검색/필터/상세 조회에 재사용

### 권장 방식
- 앱 시작 시 한 번 로드한다.
- 서비스 내부에서 캐시된 데이터를 재사용한다.
- 원본 JSON 파일은 수정하지 않는다.

## 10. 요청 처리 흐름

### 공통 흐름
1. 클라이언트가 요청한다.
2. Router가 요청을 받는다.
3. Router가 Service를 호출한다.
4. Service가 DB 또는 JSON 로더를 사용한다.
5. Service가 결과를 반환한다.
6. Router가 응답 스키마로 변환해 클라이언트에 반환한다.

### 간단한 도식
```text
Client -> Router -> Service -> DB/JSON -> Service -> Router -> Client
```

---

## 11. 게시글 CRUD 흐름

### 생성
1. POST /api/posts 요청 수신
2. Router가 request body 검증
3. Service가 `Post` 객체 생성
4. DB에 저장
5. 생성된 게시글 응답 반환

### 조회
1. GET /api/posts 또는 GET /api/posts/{id}
2. Service가 DB에서 게시글 조회
3. 응답 스키마로 변환

### 수정
1. PUT /api/posts/{id} 요청 수신
2. 기존 게시글 조회
3. 비밀번호 비교
4. 내용 갱신
5. 저장 후 응답 반환

### 삭제
1. DELETE /api/posts/{id} 요청 수신
2. 존재 여부 확인
3. 비밀번호 비교
4. DB에서 삭제
5. 성공 응답 반환

---

## 12. 통합 검색 흐름

1. GET /api/search 요청 수신
2. Router가 `query`, `category`를 전달
3. Service가 JSON 로더에서 지역정보 검색
4. Service가 DB에서 게시글 검색
5. 두 결과를 통합해 `result_type`으로 구분
6. 응답 반환

### 검색 우선순위
- 지역정보는 JSON 데이터 기준으로 검색
- 게시글은 SQLite 기준으로 검색
- 결과는 `location` 또는 `post`로 구분

---

## 13. 챗봇 흐름

1. POST /api/chat 요청 수신
2. Router가 `message`를 Service로 전달
3. Service가 질문 유형 분류
4. 검색 Service를 이용해 관련 데이터 조회
5. 답변 문장 생성
6. `answer`와 `references`를 응답

### 구현 범위
- 외부 AI 모델 호출은 제외한다.
- 규칙 기반 응답으로 구현한다.

---

## 14. 의존관계

### 핵심 의존관계
- Router는 Service에 의존한다.
- Service는 DB 세션과 JSON 로더에 의존한다.
- Service는 Pydantic Schema를 사용해 응답 형식을 맞춘다.
- `main.py`는 Router를 등록한다.

### 의존성 방향
```text
Router -> Service -> Database/JSON/Model
```

## 17. 피해야 할 과도한 구조

다음 구조는 이번 프로젝트 범위에서는 과도하다.

- Repository 패턴을 너무 세분화한 구조
- Dependency Injection을 과하게 분산한 구조
- Domain layer, UseCase layer, CQRS 같은 지나친 계층 분리
- Redis, Celery, Docker Compose 같은 추가 인프라
- 외부 AI 서비스 연동용 추상화 계층
- 사용자 인증/권한, 댓글/좋아요 등 미필수 기능용 모듈

### 이유
- 납기와 구현 난이도를 고려할 때, 현재 요구사항에 비해 복잡도가 너무 높다.
- 초보자도 이해하고 빠르게 구현할 수 있는 최소 구조가 더 적합하다.

---

## 18. 시퀀스다이어그램과 다른 구조를 제안하는 이유

시퀀스다이어그램은 사용자 흐름 중심이지만, 백엔드 아키텍처 문서는 책임 분리 중심으로 설계해야 한다. 따라서 다음과 같이 구조를 나누는 것이 적합하다.

- Router: HTTP 요청 처리
- Service: 비즈니스 로직
- Model: DB 구조
- Schema: API 형식
- Database/Config: 환경 및 DB 연결

이 구분은 시퀀스다이어그램에서 보이는 흐름을 코드 구조로 자연스럽게 옮길 수 있다.

---

## 19. 구현 시 주의사항

- 지역정보는 JSON 로더를 통해 읽고 메모리에 캐시한다.
- 게시글은 SQLite에만 저장한다.
- 비밀번호는 응답에서 제외한다.
- JSON 원본은 수정하지 않는다.
- .env 파일을 사용해 환경변수를 분리한다.
- 기능을 한 번에 너무 많이 추가하지 않고, 단계별로 구현한다.
